# TA选课系统备份策略说明 v1

## 1. 目标

当前 TA 选课系统使用：

- SQLite 数据库
- 本地上传附件目录
- 本地 `.env.local` 配置文件

因此备份策略的核心目标是：

1. 防止数据库损坏或误删
2. 防止上传附件丢失
3. 防止配置文件丢失导致邮件或访问异常
4. 保证出现问题后可以快速恢复

## 3. 当前推荐备份频率

### 3.1 数据库

建议：

- 每日备份 1 次
- 在大批量导入前额外备份 1 次
- 在版本升级前额外备份 1 次

### 3.2 上传目录

建议：

- 每日备份 1 次
- 若简历上传频繁，可每 12 小时备份 1 次

### 3.3 `.env.local`

建议：

- 每次修改后立即手工备份

### 3.4 代码

建议：

- 以 GitHub 为代码版本备份

说明：

- 代码主要依赖 Git 管理
- 运行数据不能只靠 Git，必须额外备份数据库与 uploads

## 4. 推荐保留周期

建议最少保留：

- 每日备份：保留 7 天
- 每周备份：保留 4 周
- 版本发布前备份：保留长期版本副本

## 5. 当前最实用的备份结构

Windows 服务器建议：

```text
D:\TASystem\backups
├── daily
│   ├── 2026-03-27
│   │   ├── ta_system_node.db
│   │   ├── uploads.zip
│   │   └── env.local.backup
├── weekly
└── releases
```

说明：

- `daily`：每天备份
- `weekly`：每周保留一个长期点
- `releases`：发版前备份

## 6. 备份优先级建议

如果时间有限，优先级如下：

1. `ta_system_node.db`
2. `uploads/`
3. `.env.local`
4. `assets/`

## 7. 什么时候必须立即备份

以下场景建议先备份再操作：

1. 教学班 Excel 导入前
2. 人员 Excel 导入前
3. 批量修改教学班开放时间前
4. 批量删除教学班前
5. 大范围测试前
6. 系统升级前
7. 切换服务器前
8. 将来切换到 MySQL 前

## 8. Windows 服务器手工备份示例

### 8.1 备份数据库

```powershell
Copy-Item D:\TASystem\app\ta_system_node.db D:\TASystem\backups\daily\ta_system_node_2026-03-27.db
```

### 8.2 备份配置文件

```powershell
Copy-Item D:\TASystem\app\.env.local D:\TASystem\backups\daily\env.local.2026-03-27.backup
```

### 8.3 压缩备份上传目录

```powershell
Compress-Archive -Path D:\TASystem\app\uploads -DestinationPath D:\TASystem\backups\daily\uploads_2026-03-27.zip -Force
```

## 9. 更稳妥的备份操作顺序

为了避免备份到半写入状态，建议顺序如下：

1. 暂停系统写入操作
2. 备份数据库文件
3. 备份 `uploads`
4. 备份 `.env.local`
5. 恢复系统正常运行

如果当前使用 PM2，可短暂停止：

```powershell
pm2 stop ta-system
```

备份完成后恢复：

```powershell
pm2 start ta-system
```

如果不能停机，也可以做热备，但当前 SQLite 模式下最稳还是短暂停机备份。

## 10. 恢复优先级

恢复时建议顺序如下：

1. 恢复 `ta_system_node.db`
2. 恢复 `uploads/`
3. 恢复 `.env.local`
4. 检查 `assets/`
5. 启动服务
6. 检查首页、登录、邮件、附件访问

## 11. 恢复后的验证清单

恢复完成后至少检查：

1. 系统首页是否能打开
2. 本地登录是否正常
3. 教学班是否能查看
4. 申请数据是否存在
5. 附件链接是否能打开
6. 邮件发送是否正常
7. 审计日志是否完整

## 12. 当前阶段建议

基于你现在仍在继续完善功能，当前最推荐的策略是：

1. 先用 SQLite 继续迭代
2. 建立“每日数据库 + uploads + `.env.local`”备份
3. 在正式上 Windows 服务器时，把这套备份策略同步过去
4. 等功能稳定、部署稳定后，再评估切 MySQL
